üå± Greenfield Test Scenario
New Project Deployment
What You're Testing

This scenario simulates deploying a brand new Java/SpringBoot application from scratch ‚Äî like starting a fresh project with no existing infrastructure or dependencies to worry about.

As a Developer, You Will
  1. Create a new service catalog entry in Cortex.io for your application
  2. Configure your repository (GitHub/GitLab) with the standard CI/CD pipeline templates
  3. Submit a deployment request through Kratix ‚Äî the platform will automatically provision your namespace on AKS
  4. Watch Harness pick up your deployment and push your SpringBoot app to the cluster
  5. Verify your app appears in Cortex with proper scorecards showing build status, security scans (Fortify/Sonatype), and code quality (SonarQube)
  6. Confirm the Leadership Dashboard reflects your new service in the portfolio view
Success Looks Like

Your new app is running on AKS, fully visible in Cortex.io with all integrations working ‚Äî scorecards populate automatically, engineering insights show your DORA metrics, and leadership can see the new service in their dashboards. Zero manual configuration required beyond your initial request.

üöÄ One-Touch Assembly ‚Üí Launch Control & Verify
Platform Engineering Workflow
Platform Engineer User Story

"As a Platform Engineer, I want to construct One-Touch Assembly and Launch Control & Verify workflows, so that we can create our first entries into the service catalog and validate our data mappings workflow through scoring."

One-Touch Assembly

Automated service onboarding workflow using Cortex's drag-and-drop block orchestrator with 200+ integration-aware components.

  1. User Input Block ‚Äî Collect service metadata (name, team, domain, owner, GitHub username) via form inputs
  2. Scaffolder Block ‚Äî Generate repository from Cookiecutter templates with boilerplate, CI/CD configs, and cortex.yaml descriptor
  3. GitHub/GitLab Block ‚Äî Create repository, configure branch protection, add team permissions, trigger initial workflow
  4. Harness Block ‚Äî Provision CI/CD pipeline from template, configure deployment stages and environments
  5. PagerDuty Block ‚Äî Create service entry, assign escalation policy, link to Cortex entity
  6. Jira Block ‚Äî Create project board, link work items to service for initiative tracking
Launch Control & Verify

Validation workflow with manual approval gates ensuring data flows correctly through to scoring.

  1. Branch Block ‚Äî Conditional paths based on service type (greenfield vs brownfield) using CEL expressions
  2. Jenkins/Harness Block ‚Äî Trigger CI build pipeline, run security scans (Fortify, Sonatype, SonarQube)
  3. Manual Approval Block ‚Äî Require sign-off from Platform Engineer before scorecard evaluation
  4. Scorecard Trigger ‚Äî Execute CQL rules against live integration data, validate Bronze level achievement
  5. Slack Block ‚Äî Send notification with score results, link to service catalog entry
  6. ServiceNow Block ‚Äî Create change record documenting service onboarding completion
Acceptance Criteria
  • One-Touch Assembly workflow successfully creates service catalog entry
  • All integration data points populate correctly in service entity
  • Launch Control executes scorecard evaluation without errors
  • Verify workflow confirms data mappings are accurate
  • Service achieves expected Bronze level score based on current CI/CD state
  • Workflow is repeatable and documented for future service onboarding

üè≠ Brownfield Test Scenario
Existing Application Migration
What You're Testing

This scenario simulates onboarding an existing Java/SpringBoot application that's already running somewhere ‚Äî like migrating a legacy app or bringing an existing service under the new platform's management.

As a Developer, You Will
  1. Register your existing application in Cortex.io ‚Äî point it to your current repository and existing deployment
  2. Connect your existing CI/CD pipelines (already in Harness) to the Kratix workflow
  3. Map your current namespace to Kratix's known namespace promise ‚Äî no disruption to running services
  4. Backfill historical data by ensuring all integrations (Fortify scans, Sonatype reports, SonarQube analysis) flow into Cortex
  5. Validate that your existing app's metrics and health data appear correctly in scorecards
  6. Verify engineering insights capture your deployment history and incident patterns
Success Looks Like

Your existing app continues running without downtime while now being fully managed by the platform. Cortex.io shows accurate data from your historical CI/CD runs, security scans populate from past Fortify/Sonatype results, and your team can finally see a unified view of all their services ‚Äî both old and new ‚Äî in one place.
